# 即兴工具箱产品文档

版本：原型完善版  
更新时间：2026-06-02  
当前原型：`prototype_complete/index.html`

## 1. 产品定位

即兴工具箱是一款面向即兴主理人、排练带领者和即兴团队成员的微信小程序。它帮助用户快速找到合适的即兴游戏，记录现场灵感和排练反馈，并把零散经验逐步沉淀为可复用的带领方法、游戏变体和团队复盘资产。

当前阶段不讨论技术架构，重点验证产品原型、信息结构、数据链路和核心交互。

## 2. 核心用户

- 即兴主理人：需要快速组织排练、选择游戏、记录反馈、沉淀带领经验。
- 排练带领者：需要在现场快速查游戏、加游戏、暂停记录、结束后复盘。
- 团队成员：需要回看团队排练记录，理解练习目标和游戏反馈。

## 3. 核心痛点

- 游戏散落在聊天、笔记和记忆里，临场找不到合适游戏。
- 排练中不方便完整记录，很多反馈结束后就忘了。
- 灵感很多，但缺少整理机制，难以变成下次可复用的方法。
- 团队排练经验无法持续积累，每次排练都像重新开始。
- 临场操作需要单手完成，页面底部输入和按钮布局要足够顺手。

## 4. 产品核心目标

- 帮用户快速找到“现在适合玩”的即兴游戏。
- 帮用户用最低成本记录灵感、游戏实践和排练过程。
- 帮用户把灵感和复盘转成可复用的个人沉淀。
- 帮团队保留排练记录，形成长期可回看的练习资产。

## 5. 核心数据对象

### 5.1 游戏

游戏是即兴游戏百科条目，包含名称、描述、人数、时长、标签、适用判断、玩法步骤、带领提示、变体、翻车点和相似游戏。

游戏支持：

- 搜索
- 标签筛选
- 状态筛选：全部、玩过、未玩过、收藏
- 列表视图和卡片视图
- 收藏
- 标记玩过
- 记录这次

### 5.2 灵感

灵感是原始输入，强调快速留下来。它可以是模糊的、未整理的、只有一句话的想法，例如“一句话交换身份可以加限制词”。

灵感支持：

- 文字输入
- 语音输入
- 选择整理方式
- 关联游戏
- 关联排练
- 保存为灵感
- 沉淀为方法卡

### 5.3 排练

排练是一次团队或个人练习，包含团队、时长、目标、游戏计划、单个游戏反馈和最终整体复盘。

排练支持：

- 快速开启排练
- 添加游戏
- 搜索游戏后加入排练
- 每个游戏标记状态：未开始、进行中、已完成
- 游戏 Keep / Try 快速记录
- 暂停排练
- 结束并进入复盘子页面

### 5.4 游戏实践反馈

游戏实践反馈是“某一次玩某个游戏”的记录，不等同于整场排练。它只关联当前游戏，可以选择是否加入排练。

包含：

- 当前游戏
- 人数
- 关联排练 / 场景
- 效果反馈
- Keep
- Try
- 下次提醒

### 5.5 个人沉淀 / 方法卡

个人沉淀是整理后的输出，强调下次可复用。它可以来自灵感、游戏反馈或排练复盘。

方法卡可包含：

- 带领提醒
- 游戏变体
- 台词想法
- 排练复盘
- 下次计划

## 6. 灵感和沉淀的区别

灵感是原始材料，解决“先别忘”的问题。它允许粗糙、不完整、未归类。

沉淀是整理后的方法，解决“下次能用”的问题。它需要有明确类型、适用场景和行动建议。

交互区别：

- 保存灵感：把当前想法保存为草稿，允许之后再整理。
- 沉淀为方法卡：把当前内容转成结构化知识，进入个人沉淀库。

当前原型已把按钮文案从“转为沉淀”改为“沉淀为方法卡”，避免和“保存灵感”混淆。

## 8. 关键弹层与弹窗

当前原型包含以下弹层：

- 筛选游戏：筛选人数、时长、目标、状态。
- 条件抽卡：抽中游戏、看详情、再抽一张、记录这次。
- 关联选择：查询并选择关联游戏或关联排练。
- 添加到排练：查询游戏并加入排练。
- 添加游戏：手动或语音录入游戏。
- 语音速记：语音记录和归类。
- 快速开启排练：团队、时长、目标、添加游戏方式。
- 记录确认：确认记录类型、关联游戏、关联排练、Keep / Try，并可沉淀为方法卡。
- 今日记录：查看今日灵感或今日排练记录。
- 个人沉淀列表：展示沉淀记录列表。
- 灵感记录列表：展示灵感记录列表。
- 记录详情悬浮卡：展示单条沉淀或灵感详情，支持左右切换。

## 9. 核心用户链路

### 9.1 找游戏并记录这次

发现页搜索或筛选游戏 -> 进入游戏详情 -> 点击记录这次 -> 进入游戏实践反馈页 -> 填写 Keep / Try / 下次提醒 -> 保存反馈 -> 记录确认 -> 可保存记录或沉淀为方法卡。

### 9.2 随机抽卡并记录

发现页点击随机浮球 -> 条件抽卡半弹窗 -> 再抽一张或记录这次 -> 进入当前抽中游戏的实践反馈页 -> 保存记录。

### 9.3 快速记录灵感

记录页点击大麦克风 -> 语音速记半弹窗开始记录 -> 选择归类到灵感 -> 完成录音 -> 存为灵感 -> 进入灵感记录页 -> 选择整理方式 -> 补充标题和内容 -> 保存灵感或沉淀为方法卡。

### 9.4 开启排练

记录页点击记录排练 -> 快速开启排练半弹窗 -> 选择团队、时长、目标和游戏添加方式 -> 进入排练记录页。

### 9.5 排练中记录游戏状态

排练记录页 -> 查看游戏计划 -> 点击状态按钮在未开始、进行中、已完成之间切换 -> 记录每个游戏的 Keep / Try -> 可继续添加游戏。

### 9.6 暂停并继续排练

排练记录页点击暂停排练 -> 返回记录页 -> 记录页显示暂停中的排练卡片 -> 点击继续 -> 回到排练记录页。

### 9.7 结束并复盘

排练记录页点击结束并复盘 -> 进入排练复盘页 -> 填写整体 Keep / Try / 下次提醒 -> 选择沉淀方向 -> 保存复盘。

### 9.8 回看个人资产

我的页 -> 点击个人沉淀或灵感记录 -> 列表半弹窗 -> 点击某条记录 -> 悬浮详情卡片 -> 左右切换上一条 / 下一条。

### 9.9 回看团队排练

我的页 -> 点击排练记录 -> 进入团队排练记录子页面 -> 查看具体排练 -> 跳转排练记录页。

## 10. 当前原型设计原则

- 保持微信小程序顶部胶囊安全区，右上角控件避免被遮挡。
- 底部只保留发现、记录、我的三大入口。
- 计划能力不独立成 Tab，而是融入排练记录链路。
- 高频操作尽量靠下，适配单手操作。
- 临场记录优先，整理沉淀后置。
- 游戏实践反馈和整场排练复盘拆开，避免数据链路混乱。
- 半弹窗用于轻量选择、录入和确认。
- 悬浮卡片用于查看单条沉淀或灵感详情。

## 13. 后续待确认问题

- 个人沉淀是否只对自己可见，还是可分享给团队。
- 团队排练记录是否需要多人共同编辑。
- 语音记录是实时转写，还是录完后生成摘要。
- 游戏库以官方种子库为主，还是用户自建为主。
- 排练中的游戏状态是否需要更细，例如跳过、临时加入、待复盘。
- 沉淀方法卡是否需要从多个记录合并生成。
- 数据是否需要区分个人空间和团队空间。

## 14. 技术选型与隔离规范

### 14.1 技术选型

- 前端采用微信小程序原生 WXML / WXSS / JS。
- 后端采用腾讯云 CloudBase。
- 为节省成本，CloudBase envId 可复用 `ImprovTool` 的同一个云开发环境。
- 新小程序工程独立放在 `wechat-cloudbase-app/`，不修改 `ImprovTool` 项目代码。

### 14.2 前后端目录

当前新工程目录：

```text
wechat-cloudbase-app/
  frontend/                  # 微信小程序原生前端
  backend/
    cloudfunctions/
      improv-api/            # 即兴工具箱专用聚合云函数
```

微信开发者工具打开 `wechat-cloudbase-app/`，其中：

- `miniprogramRoot` 指向 `frontend/`
- `cloudfunctionRoot` 指向 `backend/cloudfunctions/`

### 14.3 与 ImprovTool 的隔离要求

虽然复用同一个 CloudBase 服务，但代码和数据必须完全隔离：

- 不修改 `ImprovTool/wechat-app` 里的前端代码。
- 不复用 `ImprovTool` 的业务云函数，例如 `live-api`、`trainer-api`、`review-api`、`participant-api`。
- 不读写 `ImprovTool` 已有数据库集合。
- 新项目所有集合统一使用 `improv_` 前缀。
- 新项目所有云存储路径统一使用 `improv/` 前缀。
- 新项目云函数统一使用 `improv-api`，通过 action 区分业务。

### 14.4 命名规范

建议集合名：

- `improv_games`
- `improv_inspirations`
- `improv_rehearsals`
- `improv_game_records`
- `improv_method_cards`

建议 action：

- `game.list`

### 14.5 编码规范

- 页面只负责展示和交互，业务请求放到 `frontend/services/`。
- 页面不得直接散落调用 `wx.cloud.callFunction`，统一经过 `services/cloud.js`。
- 所有云函数返回统一结构：`code`、`message`、`data`、`requestId`。
- 所有写入由服务端补充 `ownerOpenId`，不要信任前端传入用户身份。
- 所有异步页面至少考虑 loading、empty、error 三态。
- 删除优先软删除，使用 `deletedAt` 标记。
- 语音输入当前先做交互占位，后续再接真实录音和转写服务。
