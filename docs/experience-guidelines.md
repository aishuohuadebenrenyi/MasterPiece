# 体验与交互规范

更新时间：2026-06-08

## 1. 体验原则

即兴工具箱是临场操作工具，不是纯内容浏览 App。设计优先级应围绕：

- 现场记录链路要短。
- 保存动作要直接。
- 整理沉淀要后置。
- 高频操作尽量靠下，适合单手完成。
- 子页面只承载深读、完整编辑和排练过程。
- 半弹窗承载轻量选择、快速输入和局部确认。

## 2. 配色语义

页面和组件配色统一从 `frontend/styles/theme.wxss` 的 `--improv-*` token 获取，不在局部 WXSS 中散落硬编码色值。

- **主题切换**：当前提供“默认”与“沉浸（theme-vivid）”两套主题。所有页面根节点（`.page`）、TabBar 和半弹窗等组件，必须能接收并正确渲染当前主题变量。
- 主动作、保存、添加等高强调入口使用 `primary` 橙色/玫瑰色体系。
- 筛选选中、次级入口等辅助强调使用 `secondary` 体系；收藏爱心使用独立的 `favorite` token（当前默认主题为奶蓝灰未收藏、奶杏粉已收藏），避免把收藏状态硬编码在组件局部。
- 页面背景、卡片、表单输入和弹层使用 `bg`、`card`、`sheet`、`field` 表面 token，**坚决避免在半弹窗或卡片中硬编码纯白色（#ffffff 或 #fff）**，以免在切换主题时出现割裂。
- 正文、弱文本、辅助说明和分割线使用 `text`、`muted`、`light`、`line` 中性色 token。
- `待同步`、云端失败但当前会话已保留等注意态使用 `warm` 体系；错误提示可使用 `error` 或 toast error token。
- 渐变、阴影、遮罩和条纹也必须使用 token，避免只抽纯色而留下不可替换的品牌色。

## 3. 临场记录规范

### 3.1 游戏反馈

默认首屏优先展示：

- 效果反馈
- Keep
- Try
- 保存反馈

人数、关联排练、下次提醒等字段可作为更多信息，不应挤占临场记录首屏。

保存反馈后应直接完成，用 Toast、轻提示或保存后入口承接“沉淀为方法卡”，避免强制二次确认。

### 3.2 灵感记录

灵感强调“先别忘”，允许粗糙和未整理。

语音速记完成后，优先保存为灵感草稿；灵感编辑页用于稍后整理，而不是保存前必经页。

### 3.3 排练记录

排练记录页主要处理过程态：

- 游戏计划。
- 游戏状态。
- 每个游戏 Keep / Try。
- 添加游戏。
- 暂停和继续。

整体复盘放到排练复盘页，避免排练过程中页面过重。

## 4. 弹层规范

半弹窗属于模态底部面板，应满足：

- 背景变暗。
- 背景不可点击。
- 背景不可滚动。
- 焦点切换到弹层。
- 底部视觉由弹层接管。

半弹窗按用途分为三类：

- 轻量半弹窗：筛选、抽卡、关联选择、语音速记，承载轻选择和局部确认，建议高度控制在 50-65vh。
- 任务型半弹窗：快速添加游戏、快速开启排练，只承载一个临场任务，建议高度控制在 70-82vh。
- 子页面替代：完整编辑、深读、长表单、排练过程和复盘，不应继续塞进半弹窗。

任务型半弹窗必须保持标题区稳定，内容区独立滚动。主按钮可按场景选择两种模式：真正需要持续可见时放入组件固定操作区；像添加游戏这类参考原型的快速表单，主按钮作为内容流最后一项，避免形成额外白色底栏。

任务型选择弹层应优先使用“搜索框 + 默认可见列表 + 确认按钮”结构，而不是把“选择”做成输入框旁边的窄按钮。记录页快速开始游戏、排练中添加游戏这类场景，打开后应直接展示可选项；搜索只作为缩小范围的辅助能力。列表必须在弹层内部独立滚动，并保留稳定高度，确保小屏设备上主按钮和底部安全区不被遮挡。

对于“添加到当前计划 / 当前上下文”一类选择弹层，列表只展示当前仍可执行的候选项，不重复展示已经加入、已经关联或当前不可再次操作的对象；如果可选项为空，应直接给出原因和下一步动作，避免用户点击后收到假成功反馈。

“添加游戏”半弹窗只做快速收录：名称和主分类为核心字段，人数、时长、标签、简短描述为推荐填写，玩法步骤默认作为可选补充项。主分类使用横向轻量 chips，标签继续作为自由输入字段，避免分类控件占用过多纵向空间。顶部提示文案应收敛，保存按钮放在表单内容末尾并居中显示。

语音按钮默认作为输入字段外侧的辅助操作，不统一内嵌到输入框内部。对多行 textarea，优先保留完整编辑空间、文本可读性和滚动区域，避免为了视觉统一压缩输入面积。

筛选弹层必须按分类分行展示，不使用所有 chips 混排的通用网格。当前筛选分类为人数、时长、目标和状态。每一类独占一行，分类标签在左，选项在右侧横向排列并允许换行。这样用户先理解筛选维度，再选择条件，避免不同维度互相挤压和视觉混乱。

实现优先级：

1. 官方模态容器能力。
2. `root-portal`。
3. 自制 fixed 遮罩。

不建议继续在各页面中堆局部样式补丁。

## 5. 底部操作栏规范

底部固定操作栏常见于详情页和表单页，必须统一处理：

- 操作栏高度稳定。
- 内容区底部 spacer 等于操作栏高度 + 安全区。
- 不出现超大空白。
- 不用大块纯白硬遮挡覆盖页面内容。
- 表单最后一个输入项必须能滚动到操作栏上方至少 24rpx。
- 主 Tab 页面底部预留应与自定义 TabBar 统一。

推荐将通用样式抽成 `fixed-action-bar` 或共享样式，而不是每页单独写。

## 6. 顶部安全区与返回按钮

微信小程序页面必须以真实胶囊位置为布局基准，而不是写死顶部 rpx：

- 返回按钮与微信右上角胶囊在同一行。
- 返回按钮只保留箭头，不显示“返回”文字。
- 返回按钮左侧应与页面内容左边界对齐。
- 除返回按钮外，子页面标题、卡片和正文顶部不能进入胶囊所在行。
- 主页面内容从胶囊行下方开始，不额外叠加一整段导航空白。

实现上通过全局布局变量承接：

- `--nav-row-top`
- `--nav-row-height`
- `--content-safe-top`
- `--page-side`
- `--page-bottom`

页面根节点必须绑定 `layoutStyle`，避免不同页面各自猜测安全区。

## 7. loading / empty / error

所有异步页面和列表至少考虑三态：

- loading：请求中展示轻量加载态，不阻塞已可用内容。
- empty：没有数据时展示下一步动作，例如添加游戏、记录灵感、开启排练。
- error：服务失败时展示可理解提示，并提供重试或当前会话内兜底。

当前阶段补充约定：

- 当前功能开发阶段不启用本地持久化缓存，页面只使用云端结果和当前会话内存态。
- 云端成功返回空数组时，必须进入 empty，不允许再回退旧本地数据。
- 云端失败时，才进入 error；如果当前会话里存在 `pending` 记录，可以继续显示这些记录。
- `mine`、`record`、`team-records` 这类聚合页，要明确区分“空数据”和“请求失败”。
- 首次无数据且页面目标明确时，应优先采用单一主动作空态，不保留多个需要数据支撑的竞争入口。
- 首次进入时，身份建立不应抢在核心任务之前；登录或授权提示应尽量绑定到真实保存、同步和回看动作。

CloudBase 写入失败时，应保留当前会话内的 `pending` 状态，避免用户输入丢失；是否做本地持久化，留待后续功能稳定后再设计。

`mine` 页中“已不再整理”属于用户的本地整理偏好，而不是业务记录本身。它可以在本地记住，避免用户每次重开小程序都重新忽略同一条待整理项；但原始灵感、游戏记录和排练复盘仍保持不删除、不回写云端。

### 7.1 极简展示规则

在当前产品定位下，页面的“简洁和优雅”主要通过信息减法实现，而不是增加装饰层。

- 零数据模块默认隐藏，不因页面平衡而强行保留 `0 条` 卡片或空白容器。
- 正式资产卡只展示已经发生的数据；尚未发生的内容更适合收敛为一张轻引导卡。
- 数字为 `0` 的统计，不默认占据视觉主位；只有进入真实积累阶段后，统计才提升为概览信息。
- 空态优先使用页面内轻卡片承接，包含标题、解释和一个下一步动作，不优先依赖 Toast 短路。
- 首屏优先保留一个主任务块和一个次级上下文块；没有真实价值的推荐、关联、扩展信息默认后置。
- 可选关联、筛选和补充说明在没有真实数据前应折叠、隐藏或弱化，不拉长首屏。
- 同一个页面中，空数据与失败状态必须视觉区分：空数据给下一步，失败给重试。

## 8. 保存与沉淀

保存和沉淀是两类动作：

- 保存：把当前记录先留下来，动作应直接。
- 沉淀：把记录整理成可复用方法卡，属于二级动作。

不应在每次保存前强制用户确认是否沉淀。更好的方式是：

- 保存成功后提供“沉淀为方法卡”的入口。
- 在我的页集中整理待沉淀记录。
- 对高价值记录给出轻提示，而不是中断式弹窗。

## 9. 轻提示规范

轻提示用于反馈低成本操作，不应遮挡正在填写的字段：

- 普通成功提示使用柔和白色半透明胶囊，例如 `已模拟语音输入`、`已加入游戏库`。
- `待同步`、云端失败但本地已保留等提示使用暖色胶囊。
- 轻提示背景、边框、文字和阴影使用 `theme.wxss` 中的 toast token，不在页面局部重写色值。
- 不使用黑色系统 toast 作为常规反馈；系统 toast 只作为拿不到当前页面实例时的兜底。
- 轻提示通过全局 `toast(title)` 触发，页面不单独创建局部提示层。
- 轻提示只承接操作结果，不承担保存确认流程。

## 10. 页面深度目标

推荐目标：

- 找游戏并开始：发现 -> 详情 -> 开始游戏 -> 结束复盘 -> 保存反馈。
- 抽卡后开始：发现 -> 抽卡 -> 详情 -> 开始游戏 -> 结束复盘 -> 保存反馈。
- 语音灵感：记录 -> 语音 -> 自动保存灵感草稿。
- 排练复盘：排练记录 -> 结束复盘 -> 保存完成。

超过两步的临场动作，应优先考虑半弹窗或保存后整理，而不是继续增加子页面。
