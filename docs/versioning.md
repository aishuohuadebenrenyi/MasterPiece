# 版本治理

更新时间：2026-06-26

本文档定义 1.0.0 正式上线后的版本号、补丁处理、兼容边界和发布验收规则。真实 AppID、envId、发布截图、CloudBase 控制台证据和操作者记录只保存在团队受控发布记录中，不提交仓库。

## 1. 版本号规则

版本号采用 `主版本.次版本.补丁版本`：

- `1.0.x`：线上补丁版本，只修线上缺陷、审核问题、安全/隐私问题和数据修复。
- `1.x.0`：兼容次版本，允许新增小功能、体验优化和帮助内容，但必须兼容 1.0.0 已有数据与接口。
- `2.0.0`：破坏性大版本，用于团队空间、多角色权限、公开分享、多端架构、数据模型重构或核心流程重构。

从 1.0.0 起，项目默认按已上线产品治理，不再随意重建生产数据、删除字段、改 action 契约或重命名已发布页面路径。

## 2. 发布后公共契约

以下内容视为发布后契约：

- CloudBase action 名称、入参语义和返回结构。
- `improv_` 集合、字段含义、权限规则和索引要求。
- 小程序页面路径、分享冷启动参数和设置页版本显示。
- 反馈记录中的版本、来源页面和选填联系方式字段。
- 前端小程序包与 `improv-api` 云函数的成对发布关系。

任何删除、重命名或语义变更都必须先判断是否影响 1.0.0 用户数据、线上客户端和微信审核材料。数据变更优先追加字段；如必须迁移，必须提供迁移脚本、回滚方式、验证样本和失败处理边界。

## 3. 版本类型

### 3.1 补丁版本 `1.0.x`

- 只处理线上 bug、审核问题、安全/隐私问题和生产数据修复。
- 不改变产品边界，不新增复杂功能，不做大规模视觉重构。
- 分支建议：`codex/hotfix-1.0.1-短描述`。
- 发布前必须复现旧问题，并证明新版本问题消失。
- 涉及云函数时，记录 CloudBase requestId、环境、操作者、时间和必要截图。

### 3.2 次版本 `1.x.0`

- 允许新增兼容能力，例如轻量设置项、帮助内容、筛选入口或非破坏性体验优化。
- 不破坏已有 action、集合字段、页面路径、分享参数和用户数据。
- 必须同步产品文档、架构/数据文档、`changelog.md` 和必要的发布运行手册。
- 发布前验证 1.0.0 已有数据仍可读取和写入。

### 3.3 大版本 `2.0.0`

- 先写 RFC 或方案文档，再进入实现。
- 必须明确旧数据迁移、失败回滚、灰度范围、用户提示和上线后监控。
- 必须验证迁移前后样本一致性，且回滚后旧版本仍能打开核心页面。
- 灰度或体验版验证完成后，才能提交正式版。

## 4. 问题流转

线上问题统一进入 `project_memory/tasks/`：

- 未开始：写入 `project_memory/tasks/todo.md`，标注发现版本、目标版本、严重级别、复现路径、影响范围和验收标准。
- 处理中：移动到 `project_memory/tasks/in-progress.md`，补充负责人、当前状态、阻塞和下一步。
- 已发布：移动到 `project_memory/tasks/done.md`，记录完成时间、发布版本、验证方式和关联文档。

正式产品事实同步写入 `docs/changelog.md`；任务流转不替代 Git 历史、微信后台发布记录或 CloudBase 部署记录。

## 5. 验收底线

补丁版本至少验证：

- 旧版本可复现，新版本不可复现。
- 修复链路不破坏素材详情、记录、练习、排练、反馈、注销中的相关核心路径。
- `npm run syntax-check` 和 `npm run typecheck` 通过；必要时运行 `npm run contract-check`。

次版本至少验证：

- 1.0.0 已有数据可继续读取和写入。
- iOS / Android 真机基础链路可用。
- 隐私政策、微信后台隐私指引和实际字段/API 一致。

大版本至少验证：

- 旧数据迁移前后样本对比通过。
- 回滚后旧版本仍可打开核心页面。
- 灰度或体验版验证完成后再提交正式版。
