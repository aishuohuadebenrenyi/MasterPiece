# 产品路线与优先级

更新时间：2026-06-26

## 发布环境验收门禁（2026-06-20）

- 在本地私有配置中绑定正式 AppID 和 develop / trial / release CloudBase 环境，不向仓库写入真实标识。
- 部署 `improv-api` 及锁文件，并在控制台应用 `backend/database_security_rules.json`。
- 微信后台线上最低基础库设为不低于 2.29.2，用非灰度基础库回归。
- 用两个账号验证数据隔离，并完成 163 条素材、超时幂等、复合事务、注销和头像清理验收。
- 真机验证 Skyline 中文输入、安全区、fixed 操作栏、3 个主 Tab 和 8 个子页。
- `wx-server-sdk@3.0.4` 的上游依赖仍有 12 个 npm audit 告警（4 moderate / 6 high / 2 critical）；强制修复会降级 SDK，发布前需跟进腾讯上游或完成风险接受。

## 1. 当前阶段判断

当前产品方向成立：当前个人版解决的是个人即兴训练中的“找素材、快记录、可沉淀”问题。当前版本可保留个人多素材 session，但不进入团队协作、团队空间、成员权限或多人共同编辑。

1.0.0 正式上线后，路线管理默认进入发布后治理：`1.0.x` 只做线上补丁，`1.x.0` 承接兼容增强，`2.0.0` 承接破坏性大改。版本号、公共契约和验收底线以 [versioning.md](versioning.md) 为准。

2026-06-14 补充：发现页已从“游戏发现”升级为“素材发现”，素材上位类型覆盖游戏、角色、才艺、格式、主理、技巧、复盘、路径。MVP 后续验证重点转为多类型素材的搜索、筛选、训练和排练编排是否稳定。

主要风险不在功能不足，而在：

- 记录链路偏深。
- 子页面和弹窗职责边界不够清晰。
- 底部 fixed 操作栏和滚动区域缺少统一规范。
- 状态层已收敛到轻量内存 store，后续重点是逐步把旧 JS 页面迁移到 TS。
- 单素材练习记录的数据颗粒度仍偏粗，暂不能支撑评分趋势、附件回看和按素材复盘归因。

## 2. P0：发布前阻塞

- 已收敛：练习复盘页底部白色遮挡和内容 spacer 不一致问题。
- 已收敛：fixed 操作栏样式和底部安全区策略，子页面返回箭头与胶囊同一行。
- 已收敛：保存反馈直接保存，移除强制记录确认弹窗。
- 已收敛：全页面常规提示改为统一轻提示，不再使用黑色系统 toast 作为常规反馈。
- 已收敛：CloudBase `improv-api` action、前端服务层和工程文档已静态核对对齐。
- 已收敛：`improv_` 集合初始化、权限和 seed 流程已在 `release-support/wechat-cloudbase-app/database.md` 和 `docs/data-api.md` 中对齐。

## 3. P1：降低临场记录成本

- 已收敛：记录页直接使用文本输入保存灵感草稿，不提供 app 内置音频采集或自动摘要。
- 已收敛：练习复盘页精简首屏字段，把人数、关联排练和下次提醒折叠到更多信息。
- 已收敛：排练复盘保存后直接完成。
- 已收敛：我的页集中承接“待整理”和“沉淀为方法卡”。
- 待实施：单素材练习记录升级为 `1-10` 分评分 + “本次复盘” + 照片/短视频附件；附件只支持查看和删除，不支持编辑。
- 待实施：素材详情页底部增加“个人练习记录”摘要，展示平均分、训练次数、最近 3 条记录和 `查看全部`；完整筛选进入练习记录页。
- 待实施：练习记录页支持按素材、评分区间、附件类型和分页查询；单条记录详情使用底部弹层或详情面板承载。

## 4. P2：信息架构收敛

- 已收敛：团队排练记录进入只读详情，而不是默认进入可编辑排练过程页。
- 已收敛：筛选弹层按人数、时长、目标、状态分行展示，不再混排。
- 已收敛：方法卡支持按来源筛选，并在待整理入口中标记来源类型。
- 已收敛当前阶段：已删除未引用的旧 `local-state.js`，页面状态来源统一回到 `store/index.ts` 和云端结果。
- 当前个人版边界：放弃分类页错落排列实验，分类页保持当前规整分类浏览；纯手势“上滑/下滑进入新页面”不进入当前版本。
- 当前个人版边界：历史 `team-records` 路径可继续存在，但产品口径解释为个人多素材 session 记录回看，不表达团队协作。

## 5. P3：增强能力

- 已收敛当前阶段：CloudBase 写入必须服务端确认后才进入历史列表；写入失败保留表单草稿并明确重试，不再使用伪 `pending` 同步语义。
- 已收敛当前阶段：团队记录页补充 loading、empty、error 三态；我的列表和今日记录明确区分空态与请求错误。
- 当前版本方向：不在 MVP 阶段提供 app 内置音频采集、转写、摘要或 AI 整理；口述输入使用系统输入法语音转文字。
- 自动推荐素材和排练模板。
- 高级统计报表。

## 6. MVP 发布差距

MVP 发布前建议至少确认：

- 素材库浏览、搜索、筛选、收藏、练过状态可用。
- 素材详情和添加素材可用。
- 素材库主路径以用户自建为主，可用示例数据完成初始化验证。
- 灵感记录可用。
- 素材练习复盘可用。
- 快速开启排练、排练素材计划、简化素材状态可用。
- 排练复盘可用。
- 我的页个人沉淀、灵感记录、排练记录可回看。
- CloudBase 数据隔离和错误提示可用。

## 7. 后置能力

以下能力不影响 MVP 基础验证，可后置：

- 多人共同编辑同一场排练。
- 团队协作、团队空间和多角色权限。
- 团队维度趋势分析和高级团队统计。
- 方法卡分享和公开传播。
- 从多个记录合并沉淀方法卡。
- AI 自动生成完整排练模板。
- app 内置音频采集、转写、自动摘要。
- 纯手势驱动的跨页面导航。
- 复杂推荐算法。

## 8. 发布后版本节奏

- `1.0.x`：处理线上缺陷、审核问题、安全/隐私问题和生产数据修复；不新增复杂功能，不改产品边界，不做大规模视觉重构。
- `1.x.0`：发布兼容小功能和体验优化；不得破坏 1.0.0 已有 action、集合字段、页面路径、分享参数和用户数据。
- `2.0.0`：承载团队空间、多角色权限、公开分享、多端架构、数据模型重构或核心流程重构；必须先写迁移方案、灰度范围和回滚边界。

## 9. 品牌与双主题落地 Roadmap

### 9.1 短期：1–2 个月

目标：让目标品牌可以安全进入产品，同时消除双主题发布阻塞。

- 第 1–2 周：确认命名迁移清单、品牌源文件、Logo 导出和灵感/现场 token；完成全部文字对比度计算。
- 第 2–4 周：按 `theme.wxss → app.wxss → 共享组件` 顺序实施双主题，不逐页硬改色值。
- 第 4–6 周：整改设置、bottom-sheet、按钮、输入、TabBar、状态面板及 VD-01/02/03/08/09/10。
- 第 6–8 周：覆盖 13 页面视觉回归、核心云端成功链路、真机中文输入、安全区和辅助技术基础测试。

交付：品牌源文件清单、双主题 token、核心组件状态矩阵、15 张以上验收截图、P0/S1 关闭记录。资源基线：设计负责人 0.5 人、前端 1 人、产品和 QA 各 0.25–0.5 人；发布配置和 CloudBase 环境由工程负责人支持。

验收：设置只出现“灵感/现场”；持久化值兼容；普通文本达到 WCAG AA；核心创建/保存/删除具备 loading+disabled；素材、练习、排练、反馈成功链路均有运行证据。

### 9.2 中期：3–6 个月

- 完成全部页面直接色值债务分类和组件状态补齐。
- 统一六个高频图标和按压/弹层/状态切换动效。
- 建立双主题截图回归、只读样式审计脚本和真机辅助技术测试清单。
- 基于用户旅程指标优化找素材、快速记录、开始练习、排练复盘和资产回看。

交付：版本化组件规范、双主题回归基线、无障碍问题清单、体验指标看板。资源：设计、前端、QA 各指定固定维护责任人，产品每月组织一次体验评审。

### 9.3 长期品牌升级

- 建立小程序、未来 H5/iOS、宣传物料和应用商店共用的品牌资产仓库。
- 每季度复审品牌识别、组件新增、主题覆盖、对比度和关键旅程指标。
- 组件和 token 使用版本记录；破坏性变化必须提供迁移清单和回滚边界。

### 9.4 设计管控流程

设计需求依次通过：品牌/语义评审 → 组件复用评审 → 灵感/现场双主题视觉走查 → 运行态与无障碍回归 → 发布后指标复盘。新增颜色、组件角色或交互模式前必须更新对应正式规范；无法提供成功链路和截图证据的项目不得标记为视觉验收完成。
