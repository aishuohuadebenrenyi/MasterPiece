# 产品架构文档

## 1. 文档目的

本文档用于沉淀当前 `ImprovTool_2 / wechat-cloudbase-app` 的实际产品技术架构，作为后续产品设计、交互整改、技术选型与实现约束的统一依据。

本文档基于当前仓库中的真实配置与代码结构整理，不基于假设性方案。

## 2. 产品形态

- 产品名称：即兴工具箱微信小程序
- 运行形态：微信小程序原生应用
- 面向对象：即兴排练、灵感记录、方法沉淀相关业务场景
- 当前端形态：多页面小程序，采用自定义 `TabBar` 组织主导航

## 3. 总体技术栈

### 3.1 前端技术栈

- 前端框架：微信小程序原生页面/组件体系
- 渲染引擎：`Skyline`
- 组件框架：`Glass-Easel`
- 语言：`TypeScript + JavaScript` 混合
- 模板：`WXML`
- 样式：`WXSS`
- UI 体系：自定义页面样式 + 自定义组件，不依赖外部大 UI 框架

### 3.2 后端与数据技术栈

- 云能力：腾讯云 `CloudBase`
- 云函数入口：`improv-api`
- 前后端通信方式：`wx.cloud.callFunction`
- 本地持久化：`wx.getStorageSync / wx.setStorageSync`

### 3.3 工程与开发工具

- 工程类型：微信小程序工程
- 开发工具：微信开发者工具
- 类型检查：`TypeScript`
- 小程序类型声明：`miniprogram-api-typings`
- 本地脚本：
  - `node tools/syntax-check.js`
  - `tsc --noEmit`

## 4. 已确认的关键配置

### 4.1 小程序渲染配置

当前项目已经明确启用 Skyline：

- `frontend/app.json`
  - `style: "v2"`
  - `renderer: "skyline"`
  - `componentFramework: "glass-easel"`
- `project.private.config.json`
  - `skylineRenderEnable: true`
- `project.config.json`
  - `libVersion: "3.16.1"`

这说明项目目标渲染路径是 Skyline，而不是默认 WebView。

### 4.2 实际运行说明

虽然项目配置已启用 Skyline，但部分页面在开发者工具中仍可能因样式或能力不兼容而回退到 WebView。

当前已识别的高风险兼容点包括：

- 某些渐变能力
- 某些 CSS Grid 写法
- 某些 scroll-snap 相关能力
- 部分偏 Web 的样式语法

因此：

- “是否启用 Skyline”是配置问题
- “当前某一页为何运行在 WebView”更多是页面实现兼容性问题

## 5. 前端架构分层

当前前端采用“小程序原生页面 + 自定义组件 + 服务层 + 轻量状态层”的分层结构。

### 5.1 分层结构

1. 应用入口层
   - `frontend/app.ts`
   - `frontend/app.json`

2. 页面层
   - `frontend/pages/discover/*`
   - `frontend/pages/record/*`
   - `frontend/pages/mine/*`
   - 以及详情、反馈、排练等子页面

3. 组件层
   - `frontend/components/*`
   - `frontend/custom-tab-bar/*`

4. 服务层
   - `frontend/services/*`

5. 状态层
   - `frontend/store/index.ts`

6. 配置与环境层
   - `frontend/config/*`
   - `project.config.json`
   - `project.private.config.json`

### 5.2 分层职责

- 应用入口层
  - 负责全局配置、云环境初始化、页面注册、渲染配置

- 页面层
  - 负责页面级状态、数据拼装、交互流程、页面跳转

- 组件层
  - 负责卡片、弹层、底部导航等可复用 UI 单元

- 服务层
  - 负责 CloudBase 云函数调用、业务接口封装、错误归一化

- 状态层
  - 负责本地轻量状态、订阅通知、部分本地持久化

## 6. UI 组件架构

### 6.1 当前 UI 策略

当前项目没有引入 TDesign、NutUI 等外部小程序 UI 框架作为主 UI 体系，而是采用：

- 页面自定义布局
- 全局样式基线
- 少量通用自定义组件

这意味着项目当前更偏向“高定制原型还原”路线，而不是“基于现成组件库快速拼装”路线。

### 6.2 关键组件

- `components/bottom-sheet`
  - 用于承载半弹窗、底部面板类交互

- `components/game-card`
  - 用于承载游戏卡片展示与事件透传

- `custom-tab-bar`
  - 用于承载主导航底栏

### 6.3 UI 选型结论

对于当前业务，不建议因为单个弹层交互问题就切换到：

- Taro
- React
- Vue

也不建议把引入大 UI 框架作为第一优先级解法。

原因：

- 当前项目已是微信原生小程序工程
- 视觉高度定制，强依赖原型还原
- 当前主要问题在模态层交互基础设施，而不是基础控件缺失

后续如果要补充基础组件库，可再评估：

- `TDesign Miniprogram`
- `NutUI MiniProgram`

但它们不是当前架构的核心前提。

## 7. 状态管理架构

### 7.1 当前状态管理方式

项目当前采用自定义轻量状态管理，而非 Redux、MobX、Pinia 一类外部状态框架。

核心文件：

- `frontend/store/index.ts`

主要机制：

- 内存态 `state`
- `subscribe()` 订阅更新
- `setState()` 局部更新
- `wx.setStorageSync()` 本地持久化

### 7.2 当前状态内容

当前 store 管理的状态主要包括：

- 视图模式：`viewMode`
- 游戏列表：`games`
- 收藏与玩过状态
- 今日灵感与今日排练
- 方法卡
- 暂停中的排练

### 7.3 当前状态层现状

项目目前处于“新轻量 store 与旧 local-state 并存”的过渡状态：

- 新页面更偏向使用 `store/index.ts`
- 部分 JS 页面仍在使用旧 `services/local-state.js`

后续架构建议是逐步统一到单一状态层，减少双轨维护成本。

## 8. 后端与接口架构

### 8.1 后端模式

当前后端采用腾讯云 `CloudBase` 云函数模式，而不是独立部署的 HTTP 服务。

统一云函数入口：

- `improv-api`

### 8.2 前端调用方式

前端通过统一封装调用云函数：

- `frontend/services/cloud.ts`

当前调用特征：

- 使用 `wx.cloud.callFunction`
- 内置超时控制
- 内置错误归一化
- 内置 requestId 生成

### 8.3 数据隔离约定

根据当前 README，项目有明确隔离边界：

- 不修改同级旧项目代码
- 不调用旧项目云函数
- 不读写旧项目数据库集合
- 统一使用 `improv-api`
- 数据集合统一使用 `improv_` 前缀

这意味着当前项目是一个相对独立的小程序业务域。

## 9. 当前交互基础设施判断

### 9.1 模态交互类型

当前业务中的半弹窗、底部卡片，不是普通浮层，而是模态底部面板交互。

这类交互的最低要求包括：

- 背景变暗
- 背景不可点击
- 背景不可滚动
- 焦点切换到弹层
- 底部视觉由弹层接管

### 9.2 当前技术判断

这类交互在微信小程序原生栈里的推荐实现优先级为：

1. 官方模态容器能力
2. `root-portal`
3. 自制 fixed 遮罩

因此，当前项目后续在弹层体系上应优先强化“模态容器基础设施”，而不是继续在页面里堆局部样式补丁。

## 10. 当前架构结论

当前项目的正式产品技术架构可归纳为：

- 客户端：微信原生小程序
- 渲染：Skyline
- 组件机制：Glass-Easel
- 视图层：WXML + WXSS
- 逻辑层：TypeScript + JavaScript
- UI 体系：自定义组件化 UI，不依赖大型外部 UI 框架
- 状态层：自定义轻量 store，部分页面仍处于旧状态实现过渡期
- 后端：CloudBase 云函数
- 调用方式：`wx.cloud.callFunction`
- 导航：自定义 `TabBar`

## 11. 后续架构建议

### 11.1 保持不变的部分

建议继续保持：

- 微信原生小程序
- Skyline
- CloudBase
- 轻量 store
- 自定义组件化 UI

### 11.2 优先优化的部分

后续优先优化方向应是：

- 清理导致 Skyline 回退 WebView 的页面样式与交互写法
- 统一弹层模态容器方案
- 统一状态层，逐步减少旧 `local-state`
- 统一页面常态与模态态下的底部空间策略

### 11.3 不建议当前阶段做的事

当前阶段不建议为了局部交互问题直接切换到：

- Taro
- React
- Vue
- 大型 UI 框架主导重构

因为这会带来较高迁移成本，但不能直接解决当前模态交互与 Skyline 兼容问题。
